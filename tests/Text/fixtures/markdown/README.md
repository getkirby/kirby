# Markdown regression fixtures

These fixtures are the **regression checks** for `Kirby\Text\Markdown`. They freeze the exact HTML the current implementation produces so any refactoring can be proven byte-for-byte identical.

## Structure

```
inputs/
  commonmark/      <section>-NNNN.md   (CommonMark spec examples, e.g. tabs-0001.md)
  parsedown/       Parsedown 1.8.0 upstream test inputs
  parsedownextra/  ParsedownExtra 0.9.0 upstream test inputs
  kirby/           Kirby's own markdown test fixtures
  <profile>/<same relative path>.html  (current pipeline output)
ATTRIBUTION.md     input provenance + third-party licenses
```

CommonMark inputs are named by their spec section and global example number, so `tabs-0001.md` is example 1 (section "Tabs") and `raw-html-0620.md` is example 620 (section "Raw HTML").

## Inputs

| Source               | Count | Notes                                                            |
| -------------------- | ----- | ---------------------------------------------------------------- |
| CommonMark spec      | 655   | spec's `→` tab markers are restored to tabs.                     |
| Parsedown 1.8.0      | 64    | `test/data/*.md` from the vendored version's upstream tag.       |
| ParsedownExtra 0.9.0 | 6     | abbreviation, (compound) footnote, definition list, attributes…  |
| Kirby's own fixtures | 2     | `markdown.md` + `inline.md`, copied from `tests/Text/fixtures/`. |

Only the markdown **inputs** are drawn from upstream. Expected HTML has _not_ been taken from upstream. It has been generated by the current Kirby pipeline, because parity is defined against _this_ Parsedown/ParsedownExtra version, not against CommonMark conformance or newer parser releases. See `ATTRIBUTION.md` for sources and licenses.

### Deliberate divergence: raw HTML blocks

One place intentionally departs from ParsedownExtra: a raw HTML block **without** a `markdown="1"` attribute is now passed through **verbatim** instead of being round-tripped through `DOMDocument`. ParsedownExtra's round-trip lower-cased tags/attributes, fabricated missing closing tags and could **drop content** (e.g. everything after the first top-level element). Skipping it keeps the author's HTML intact (and is much faster). Blocks carrying `markdown="1"` still use the DOM path to re-parse their content. A handful of `commonmark/html-blocks-*`, `autolinks-*`, `link-reference-definitions-*` and `parsedown/markup_consecutive_*` snapshots reflect this.

## Profiles (option combinations)

Each input has been rendered under every profile, producing a separate snapshot per combination. Kirby's defaults are `breaks=true, safe=false`; `inline` is the `parse()` argument.

| Profile    | breaks | safe | inline |
| ---------- | :----: | :--: | :----: |
| `default`  |   ✓    |      |        |
| `nobreaks` |        |      |        |
| `safe`     |   ✓    |  ✓   |        |
| `inline`   |   ✓    |      |   ✓    |

The `safe` profile pins the safe-mode escaping behavior, which is both an output-parity requirement and an XSS boundary (HTML escaping and `javascript:`/`data:` URL neutralisation).

## Known issues

None. `commonmark/raw-html-0620.md` (`<33> <__>`, invalid HTML tag names) previously threw a fatal error under the unsafe `default` / `nobreaks` profiles because the markup was fed to `DOMDocument`. Now that HTML blocks without `markdown="1"` bypass the DOM entirely, it renders verbatim, so `MarkdownTest::KNOWN_CRASHES` is empty.
